Transmittal Note

This packet responds to the HIPAA Legal Memorandum provided by counsel and documents PPN Portal's current privacy architecture, HIPAA compliance posture, and the status of identified conflicts. All findings are based on live schema inspection, direct engineering review, and practitioner/clinical expert input.

PPN Portal is architected as a Zero-PHI documentation platform. No Protected Health Information is transmitted to, stored in, or processed by PPN's systems. All patient data is pseudonymized at the practitioner's device prior to submission. This packet provides the technical evidence supporting that position.

Important: This packet is prepared for legal counsel review. The posture overview (Section 1) represents the product's intended design, not a legal opinion. Counsel should independently evaluate all positions stated herein.

What PPN Portal Is

PPN Portal is a clinical documentation and network intelligence platform for psychedelic therapy practitioners. It enables practitioners to log structured clinical data, benchmark outcomes, and generate compliance-ready exports, without ever transmitting, storing, or processing Protected Health Information (PHI) as defined under HIPAA.

1. No PHI Is Collected or Stored

PPN Portal is architected from the ground up to eliminate PHI at the point of entry. The following identifiers are never collected:

Patient Name
No direct identifier stored
Date of Birth
Age stored as range only, not exact date
Social Security Number
Not captured, not relevant
Patient Contact Info
Practitioners manage their own patient contact
Medical Record Number
No MRN field in any database table
ICD-10 Diagnosis Codes
Not collected
Insurance Information
Not collected
Biometric Identifiers
Not collected

2. Pseudonymization by Design (Phantom Shield)

Every patient record is identified only by a system-generated anonymous code in the format PT-XXXXXXXXXX, produced by a cryptographically random algorithm using an unambiguous 32-character alphabet. This code:

  • Cannot be reverse-engineered into a real patient identity. It contains no initials, no birthdate, no clinic name, and no MRN.
  • Is not derived from any patient attribute. The generation algorithm is confirmed fully random. No practitioner input of any kind influences the generated code.
  • Is generated locally on the practitioner's device at session start, before any data reaches PPN servers.
  • Is stored only as a one-way hash in PPN's database. PPN never sees or retains the plaintext code in any analytics-facing table.

The practitioner's own clinic retains the only mapping between the anonymous code and the real patient identity. PPN is not party to that mapping.

3. Structured, Controlled Inputs Only

No clinical form in PPN Portal accepts free-text patient identifiers. All clinical inputs use structured, fixed-vocabulary controls: dropdown selections, sliders, checkboxes, and foreign-key lookups to controlled vocabulary tables.

4. Access Controls

  • All database tables enforce Row Level Security (RLS). Practitioners can only access records associated with their own clinical site.
  • No patient record is visible to a different practitioner's account, by architectural constraint, not policy alone.
  • Production database access is restricted to authorized administrators only. Engineering does not directly query production records. Records contain no PHI.

Legal Position Under HIPAA

QuestionPosition
Covered Entity?No. PPN Portal is not a health plan, healthcare clearinghouse, or healthcare provider. Not a Covered Entity under 45 CFR §160.103.
Business Associate?No. A BAA requires a Covered Entity to disclose PHI to us. Because pseudonymization occurs at the practitioner's device before data reaches our servers, no PHI is disclosed. No BAA is required under this design.
Safe Harbor target?Yes. Design targets HIPAA Privacy Rule Safe Harbor method (45 CFR §164.514(b)(2)), which permits de-identified data to be used and disclosed freely.

3. Pseudonymization Specification

Code Generation Algorithm

Reviewed and confirmed, March 2026.

Algorithm Specification

Patient codes are generated as PT-XXXXXXXXXX using a 32-character unambiguous alphabet (A-Z excluding I and O; 0-9 excluding 0 and 1). 10 characters = approximately 3.1 x 1014 unique values. Generation is performed entirely client-side on the practitioner's device. No patient attribute (name, DOB, MRN, initials) is used as input. PPN servers never receive the plaintext code.

Hash Storage

The plaintext code is hashed before transmission. Analytics and clinical output tables store only the one-way hash of the patient code. The plaintext is retained only in the practitioner-facing administrative record, which is protected by Row Level Security and accessible only by the practitioner who created it.

Camera Scanner Alternative

Practitioners may also generate patient IDs by scanning a wristband QR code via the platform's camera scanner. No patient name is entered at any step. The scanner inputs the generated code directly, bypassing the keyboard entirely.

No analytics tools (Posthog, Google Analytics, Mixpanel), no error tracking tools (Sentry, Datadog), no observability tools, and no additional integrations are configured. Confirmed via environment variable review and vendor integration panel review.

5. Authentication and Account Model

Authentication is handled entirely by Supabase Auth. No passwords are stored by PPN. Email-based magic link authentication is the primary method. Practitioners have individual accounts tied to one or more clinical sites via a practitioner-site assignment table. RLS policies enforce site-level data isolation, so a practitioner cannot access another site's records.

6. Backup and Disaster Recovery

Supabase manages point-in-time recovery (PITR) for the PostgreSQL database. The current Supabase plan includes daily backups with a 7-day retention window. No internal PPN backup infrastructure exists, this is fully delegated to Supabase. PPN does not maintain a separate backup of patient records.
