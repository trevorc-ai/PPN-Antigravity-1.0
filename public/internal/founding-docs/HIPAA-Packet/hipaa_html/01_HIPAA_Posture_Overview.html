What PPN Portal Is

PPN Portal is a clinical documentation and network intelligence platform for psychedelic therapy practitioners. It enables practitioners to log structured clinical data, benchmark outcomes, and generate compliance-ready exports, without ever transmitting, storing, or processing Protected Health Information (PHI) as defined under HIPAA.

1. No PHI Is Collected or Stored

PPN Portal is architected from the ground up to eliminate PHI at the point of entry. The following identifiers are never collected:

Patient Name
No direct identifier stored
Date of Birth
Age stored as range only, not exact date
Social Security Number
Not captured, not relevant
Patient Contact Info
Practitioners manage their own patient contact
Medical Record Number
No MRN field in any database table
ICD-10 Diagnosis Codes
Not collected
Insurance Information
Not collected
Biometric Identifiers
Not collected

2. Pseudonymization by Design (Phantom Shield)

Every patient record is identified only by a system-generated anonymous code in the format PT-XXXXXXXXXX, produced by a cryptographically random algorithm using an unambiguous 32-character alphabet. This code:

  • Cannot be reverse-engineered into a real patient identity. It contains no initials, no birthdate, no clinic name, and no MRN.
  • Is not derived from any patient attribute. The generation algorithm is confirmed fully random. No practitioner input of any kind influences the generated code.
  • Is generated locally on the practitioner's device at session start, before any data reaches PPN servers.
  • Is stored only as a one-way hash in PPN's database. PPN never sees or retains the plaintext code in any analytics-facing table.

The practitioner's own clinic retains the only mapping between the anonymous code and the real patient identity. PPN is not party to that mapping.

3. Structured, Controlled Inputs Only

No clinical form in PPN Portal accepts free-text patient identifiers. All clinical inputs use structured, fixed-vocabulary controls: dropdown selections, sliders, checkboxes, and foreign-key lookups to controlled vocabulary tables. Screenshots confirming absence of free-text patient name fields are available as supporting exhibits.

4. Access Controls

  • All database tables enforce Row Level Security (RLS). Practitioners can only access records associated with their own clinical site.
  • No patient record is visible to a different practitioner's account, by architectural constraint, not policy alone.
  • Production database access is restricted to authorized administrators only. Engineering does not directly query production records. Records contain no PHI.

Legal Position Under HIPAA

QuestionPosition
Covered Entity?No. PPN Portal is not a health plan, healthcare clearinghouse, or healthcare provider. Not a Covered Entity under 45 CFR §160.103.
Business Associate?No. A BAA requires a Covered Entity to disclose PHI to us. Because pseudonymization occurs at the practitioner's device before data reaches our servers, no PHI is disclosed. No BAA is required under this design.
Safe Harbor target?Yes. Design targets HIPAA Privacy Rule Safe Harbor method (45 CFR §164.514(b)(2)), which permits de-identified data to be used and disclosed freely.
Counsel Note

The positions above represent the product's intended design posture, not a legal opinion. This document requires independent counsel review and validation against the specific fact pattern of how practitioners use the platform in practice.
