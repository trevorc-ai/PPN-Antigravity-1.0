C1 — Exact Age Storage Under Remediation

What the Memo Required

HIPAA Safe Harbor (45 CFR §164.514(b)) prohibits storing ages for individuals 90 and older as an exact integer. Best practice for a multi-practitioner analytics platform is to store age as a bracket/range. The public PrivacyPolicy.tsx page claims ages are "generalized into ranges." The schema currently stores an exact integer.

Affected Fields

log_clinical_records.patient_age_years (integer) and log_patient_profiles.age_at_intake (integer).

Resolution Path

Add a ref_age_ranges lookup table (e.g., 18-24, 25-34, 35-44 ... 90+) and replace exact integer columns with a foreign-key reference. This mirrors the ref_weight_ranges pattern already implemented. Work order filed (WO-655 equivalent for age). Privacy Policy text must not be updated until this fix is deployed.

C2 — Exact Timestamp Storage Under Remediation

What the Memo Required

HIPAA Safe Harbor prohibits dates more specific than the year for any element that can re-identify a subject. Session-level timestamp fields are an area of concern.

Affected Fields

submitted_at, dose_administered_at, onset_reported_at, peak_intensity_at, session_ended_at, session_started_at, meq30_completed_at, edi_completed_at, ceq_completed_at, contraindication_assessed_at in log_clinical_records; assessment_date in log_patient_assessments.

Resolution Path — Two Options (Counsel Must Choose)

Option A (Expert Determination): Engage a qualified statistician to certify that re-identification risk is sufficiently small. This is the only path that allows retention of exact timestamps in research exports.

Option B (Safe Harbor): Replace direct timestamp queries in all analytics and export views with year-only derived fields. Raw timestamps may be retained in operational tables for session management, but must be suppressed in all de-identified outputs.

C3 — Plaintext Patient Link Code Defensible — Documented

Architecture

The schema retains the plaintext patient_link_code in log_patient_site_links (practitioner-facing administrative record) alongside the one-way hash patient_link_code_hash in log_clinical_records (all analytics and clinical outputs).

Defensibility Conditions (All Must Hold)

1. log_patient_site_links is protected by RLS — only the practitioner who created the link can access it.
2. The plaintext code is never exported in any de-identified dataset.
3. The code is fully random and non-derived from any patient attribute. Confirmed: generatePatientId() in src/services/identity.ts uses a 32-character unambiguous alphabet with no PHI embedding.
4. Cross-site linkage via transferred_from_site_id is possible but limited to practitioner-authorized transfers — no automated cross-site data sharing occurs.
