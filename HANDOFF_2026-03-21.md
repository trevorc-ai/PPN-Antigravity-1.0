## QA Results — What FAILED After Track A Commits

**Test environment:** Live mobile view at `ppnportal.net/#/wellness-journey` with active dosing session.

### FAIL 1: Session Vitals Trend chart completely blank
- **Symptom:** Panel opens, legend chips present, chart area completely empty. No data plotted at all.
- **Root cause:** `h-full min-h-[220px]` fix to `SessionVitalsTrendChart` changed the chart container but the chart itself (`ResponsiveContainer`) needs a parent with a resolved pixel height. `h-full` in a flex chain that isn't closed with a known pixel height renders as 0px in some browser contexts. Also: `vitalsChartData` in the cockpit comes from `updateLog` which is populated by `handleSaveUpdate` — but the cockpit's `SessionCockpitView` passes `vitalsChartData` from `DosingSessionPhase` which may not be receiving the data from `SessionCockpitView`'s own update log. **The cockpit and the chart are not sharing state properly.**
- **Desktop view (screenshot 5):** Chart is also empty but the component renders. Vitals data IS in `log_session_vitals` — seen via the HUD (HR 62 bpm, BP 122/82). The chart just isn't plotting it.

### FAIL 2: Live Session Timeline — no real-time updates (requires page refresh)
- **Symptom:** New events (Session Update, Music, Decision, Observation, Rescue Protocol) do NOT appear in the cockpit timeline until the user manually refreshes the browser.
- **Root cause confirmed:** `DosingSessionPhase.tsx` dispatches `ppn:dose-registered` for Additional Dose events (line 364, 1047) — which is why ADD DOSE appears immediately. But ALL other action buttons (Session Update, Rescue Protocol, Music, Decision, P.Spoke, Adverse Event) do NOT dispatch any custom event that `LiveSessionTimeline` listens to. The 60-second poll that covered this was removed (correctly — it caused the page-scroll bug) but no replacement event bus was added for these actions.

### FAIL 3: Vital Signs missing from SESSION UPDATE entries in timeline
- **Symptom:** `[SESSION UPDATE]` entries appear in the timeline after refresh but show "Session update logged" with no vitals detail — even when HR/BP were entered.
- **Root cause:** `handleSaveUpdate()` in `DosingSessionPhase` calls `createTimelineEvent` with `metadata: { event_description: 'Session update logged' }`. It does NOT include vitals in the description or metadata. The vitals are written separately to `log_session_vitals` table but never linked to the timeline event text.

### FAIL 4: Rescue Protocol — does not appear in timeline at all (even after refresh)
- **Symptom:** After clicking Rescue Protocol and submitting, no entry appears in the timeline before or after refresh.
- **Root cause:** Needs investigation. The rescue protocol handler likely writes to a different table or triggers a different flow that never creates a `log_flow_events` row for this session.

### FAIL 5: Observation, Music, Decision — appear after refresh but with no details
- **Symptom:** `[OBSERVATION] Patient reported`, `[MUSIC] Playlist changed to`, `[DECISION] Decision made` — all appear with generic placeholder text, not the actual values entered.
- **Root cause:** The `handleAddNote` call for these quick-action chips passes only `presetDesc` as a hardcoded generic string (e.g., `'Patient reported: '` or `'Playlist changed to'`). The actual detail value entered by the practitioner is never appended.

### FAIL 6: Adverse Event — incorrect timestamp
- **Symptom:** `[ADVERSE EVENT]` appears after refresh with correct details but wrong timestamp (shows 4:02 PM for an event clearly logged at ~9 AM session time).
- **Root cause:** The timestamp stored in `event_timestamp` for adverse event entries is likely being captured from a different clock source or formatted incorrectly before being passed to `createTimelineEvent`.

## Root Cause Summary — The Structural Issue

The desktop view (screenshot 5) and the mobile cockpit view are **two different component trees** that have diverged:

- **Desktop full view:** Uses the old `DosingSessionPhase` → `LiveSessionTimeline` stack with the complete event bus. The timeline shows 20+ events correctly including all event types with full details. This view works.

- **Mobile cockpit view:** Uses `SessionCockpitView` → `LiveSessionTimeline` (same component) BUT `DosingSessionPhase` only dispatches `ppn:dose-registered` to the event bus for dose events. The 6 other action types are fire-and-forget to the DB with no event bus notification.

**This is exactly Track B's scope.** Track B is not just about `ActivePatientContext` routing — it is about establishing a **single event bus** (`ppn:session-event` or equivalent) that all cockpit actions dispatch to, so `LiveSessionTimeline` receives real-time updates without polling.

## Track B — Revised Scope (What the Next Agent Must Implement)

The STABILIZATION_BRIEF.md Track B items stand as written, but the implementation approach must address:

### B0 (NEW — prerequisite): Unified Session Event Bus
**File:** `DosingSessionPhase.tsx` and specifically `handleSaveUpdate()`, the Rescue Protocol handler, and quick-action handlers.

Every cockpit action that creates a timeline entry must dispatch:
```typescript
window.dispatchEvent(new CustomEvent('ppn:session-event', {
  detail: { type: string, label: string, timestamp: string }
}));
```

`LiveSessionTimeline` already listens for `ppn:session-event` (line 432). `ADD DOSE` already fires it correctly. The other 5 action types do not. This is a surgical fix to `DosingSessionPhase.tsx` only.

### B0b (NEW): Enrich Session Update timeline entries with actual vitals
In `handleSaveUpdate()`, when `updateHR` or `updateBP` are entered, include them in the timeline description:
```
"Session update logged | HR: 72 bpm | BP: 120/80"
```
This means the `createTimelineEvent` call must build its description string from the captured vitals fields, not a hardcoded generic string.

### B0c (NEW): Fix vitals chart — use fixed pixel height in cockpit context
`SessionVitalsTrendChart` with `h-full` relies on a resolved parent height. In the cockpit, the parent is `h-[380px]` → wrapping `h-full div` → `SessionVitalsTrendChart` root. The `ResponsiveContainer` needs a concrete height. Change the chart wrapper inside `SessionVitalsTrendChart` to use `height="100%"` only when parent height is resolved, OR pass an explicit `chartHeight` prop from the cockpit (e.g., `chartHeight={340}`).

### B1–B5: As written in STABILIZATION_BRIEF.md
- B1: `ActivePatientContext` + URL param routing
- B2: Session Prep view wiring
- B3: Session closeout + `endDosingSession` wiring
